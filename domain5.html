Task 5.1: Context Window 三大陷阱 — 你的資訊是怎麼消失的

  "order_id": "ORD-123",
  "customer_id": "C-456",
  "billing_address": "...",
  "shipping_address": "...",
  "payment_method": "...",
  "payment_last4": "...",

// PostToolUse hook 修剪 output

  "status": "delivered",
  "total": 89.99,

// 只保留退貨相關的 5 個欄位
// 省下 80% 的 token
// Claude 不會被無關資訊干擾

// 額外策略：
// 1. 關鍵事實放到 prompt 開頭
// 2. 用 section header 組織內容
// 3. 數字/日期抽到 "case facts" 區塊
Python Case Facts 持久化技巧 ▼ 
# 把交易事實抽到獨立區塊，每次 prompt 都包含
# 避免 progressive summarization 把數字模糊化

case_facts = """
=== CASE FACTS (DO NOT SUMMARIZE) ===
Customer: C-456 (Jane Chen, Premium member)
Order: ORD-123, delivered 2026-03-15
Total: $89.99 (2 items)

# → 不管對話多長，關鍵數字永遠不會被摘要掉
Task 5.2

Task 5.2: Escalation — 什麼時候該轉人工？

客戶說了什麼？
→ 立即 escalate。
客戶明確要求轉人工 → 不要嘗試調查，不要試圖解決，直接轉接。
這是最優先的 escalation 觸發條件。
→ Escalate（政策缺口）。
政策只寫了「自家價格調整」，沒寫比價。政策模糊或沈默的情況 → agent 不該猜測，應該 escalate。
讓人類做政策判斷。
→ 自主處理。
這是標準的損壞退貨流程，有照片證據，在 agent 能力範圍內。
Escalate 這種簡單案例 = 浪費人工資源，降低首次解決率。
→ 先嘗試解決，被拒絕才 escalate。
承認客戶的挫折感，同時提供解決方案。
如果客戶再次堅持要轉人工 → 那時才 escalate。
注意：情緒 ≠ 複雜度。不要用情緒分析來觸發 escalation。
考試陷阱：「讓 agent 自報信心分數 1-10」是錯誤答案。LLM 自報信心校準很差 — 在困難案例上反而過度自信。正確做法：明確標準 + few-shot 範例。
Task 5.3

Task 5.3: 跨 Agent 錯誤傳播 — 別壓抑也別崩潰

情境思考：subagent 搜尋 timeout 了，coordinator 該怎麼辦？

Web search subagent 在搜尋中 timeout。你要設計錯誤回傳方式。哪個選項讓 coordinator 能做最好的決策？

A — 結構化錯誤 context。Coordinator 拿到這些資訊後可以：
- 用修改後的查詢重試 ✓
- 嘗試替代方案 ✓
- 用部分結果繼續並標注覆蓋缺口 ✓

B 的「搜尋不可用」隱藏了所有 context，coordinator 變瞎子。
C 壓抑錯誤 — 最危險的反模式，coordinator 以為成功了但資料不完整。
D 一個 subagent 失敗就整個掛 — 太脆弱。

Task 5.5 & 5.6

Task 5.5–5.6: 信心校準和來源追溯

情境思考：你的抽取系統 97% 準確率，可以自動化嗎？

你的文件抽取系統整體準確率 97%。老闆說「夠好了，自動化吧」。你應該怎麼回應？

B — 聚合指標會掩蓋問題。

97% overall 可能是：發票 99.5% + 合約 99% + 手寫表格 70%。手寫表格拉低了整體，但被其他類型稀釋了。

正確步驟：
1. 按文件類型和欄位分析準確率
2. 對高信心抽取做分層隨機抽樣（不是只看低信心的）
3. 全部段都一致後才自動化
4. 持續監控，偵測新型錯誤模式

C 不夠：只路由低信心的會漏掉高信心但錯誤的案例。
來源追溯（Provenance）速記：
1. Subagent 輸出必須包含 claim-source mapping（來源 URL、文件名、摘錄）
2. 衝突的統計數據 → 保留兩個值 + 來源標注，不要隨意選一個
3. 要求包含 出版/收集日期，防止時間差異被誤判為矛盾
4. 不同內容類型用不同格式呈現（財務 → 表格、新聞 → 散文）
