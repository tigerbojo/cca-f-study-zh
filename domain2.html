Domain 2: 工具設計與 MCP 整合

佔分 18%
核心考點：Tool description 是 LLM 選擇工具的主要依據。模糊或重疊的描述會導致 misrouting。MCP 的 isError flag、結構化錯誤回應、以及工具分配策略是本章重點。

Task 2.1: 設計有效的 Tool 介面 — 清晰描述與邊界

必考知識

  • Tool descriptions 是 LLM 選擇工具的主要機制；描述太簡略 → 選擇不可靠
  • 描述應包含：輸入格式、範例查詢、邊界案例、何時用它 vs 類似工具
  • 模糊/重疊描述會導致 misrouting（如 analyze_content vs analyze_document 描述幾乎一樣）
  • System prompt 中的關鍵字敏感指令可能覆蓋好的 tool description

必考技能

  • 寫出清楚區分每個工具的描述：用途、預期輸入/輸出、何時用 vs 類似替代
  • 重命名工具消除功能重疊（如 analyze_content → extract_web_results）
  • 拆分通用工具為特定用途工具（如 analyze_document → extract_data_points + summarize_content + verify_claim_against_source）

Task 2.2: 實作 MCP 工具的結構化錯誤回應

必考知識

  • MCP isError flag：向 agent 傳達工具失敗
  • 錯誤分類：暫時性（timeout）、驗證（無效輸入）、業務（政策違規）、權限
  • 統一的泛用錯誤（"Operation failed"）阻止 agent 做出適當的恢復決策
  • 可重試 vs 不可重試錯誤：結構化 metadata 防止浪費重試

必考技能

  • 回傳結構化錯誤 metadata：errorCategory、isRetryable boolean、人類可讀描述
  • 業務規則違規：retriable: false + 客戶友善說明
  • Subagent 內部處理暫時性錯誤，只傳播無法本地解決的錯誤（附帶部分結果和嘗試記錄）
  • 區分存取失敗（需重試決策）vs 合法空結果（查詢成功但沒有匹配）

Task 2.3: 適當分配工具給各 Agent 並設定 tool_choice

必考知識

  • 給 agent 太多工具（如 18 個而非 4-5 個）會降低工具選擇可靠性
  • 擁有職責外工具的 agent 傾向誤用它們（如 synthesis agent 嘗試 web search）
  • Scoped tool access：只給 agent 其角色需要的工具
  • tool_choice 選項："auto"、"any"、強制指定工具

必考技能

  • 限制每個 subagent 只有其角色相關的工具
  • 用受限替代工具取代通用工具（如 fetch_url → load_document）
  • 為高頻需求提供 scoped 跨角色工具（如 synthesis agent 的 verify_fact）
  • tool_choice 強制選擇：確保特定工具先被呼叫（如強制 extract_metadata）
  • tool_choice: "any"：保證模型呼叫工具而非回傳對話文字
考試陷阱：當 agent 經常選錯工具時，「第一步」通常是改善 tool description（低成本高效果），而非加 few-shot 範例（增加 token overhead）或建 routing layer（過度工程化）。

Task 2.4: 將 MCP Server 整合到 Claude Code 和 Agent 工作流

必考知識

  • MCP Server scope：專案級 .mcp.json（團隊共享）vs 用戶級 ~/.claude.json（個人）
  • .mcp.json 中的環境變數展開（如 ${GITHUB_TOKEN}）— 不用 commit 密鑰
  • 所有設定的 MCP server 在連線時被發現，工具同時可用
  • MCP resources：暴露內容目錄（issue 摘要、文件層級、DB schema）減少探索性 tool call

必考技能

  • 在 .mcp.json（專案級）設定共享 MCP server + 環境變數展開
  • 在 ~/.claude.json（用戶級）設定個人/實驗性 MCP server
  • 增強 MCP tool description，防止 agent 偏好內建工具（如 Grep）而忽略更強大的 MCP 工具
  • 優先用現有社群 MCP server（如 Jira），只在團隊特殊需求時才建自訂 server
  • 將內容目錄暴露為 MCP resources，讓 agent 知道有哪些資料可用

Task 2.5: 有效選用內建工具（Read, Write, Edit, Bash, Grep, Glob）

必考知識

  • Grep：內容搜尋（搜尋檔案內容中的 pattern，如函式名、錯誤訊息）
  • Glob：檔案路徑模式匹配（依名稱或副檔名找檔案）
  • Read/Write：完整檔案操作；Edit：用唯一文字匹配做精準修改
  • 當 Edit 因非唯一文字失敗時，用 Read + Write 作為 fallback

必考技能

  • 搜尋程式碼內容 → Grep（找函式呼叫者、定位錯誤訊息）
  • 找檔案 pattern → Glob（如 **/*.test.tsx）
  • 漸進式理解 codebase：先 Grep 找進入點 → Read 追蹤 import → 而非一開始就讀所有檔案
  • 追蹤跨 wrapper module 的函式使用：先找出所有 exported names → 再搜尋每個名稱
