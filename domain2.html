Task 2.1

Task 2.1: Tool Description 怎麼寫才不會讓 Claude 選錯工具？

Tool description 是 LLM 選擇工具的主要依據。寫不好 → Claude 會搞混相似的工具。

✗ 爛的 Tool Description
{
  "name": "get_customer",
  "description": "取得客戶資訊"
}

{
  "name": "lookup_order",
  "description": "取得訂單詳情"
}

// 問題：描述幾乎一樣
// Claude 看到 "查我的訂單 #123"
// 不確定該用哪個 → 60% 選錯
✓ 好的 Tool Description
{
  "name": "get_customer",
  "description": "用 email 或電話
    查詢客戶帳戶資訊。回傳
    customer_id、姓名、會員等級。
    ⚠️ 不接受訂單號碼。
    用途：身份驗證、帳戶查詢"
}

{
  "name": "lookup_order",
  "description": "用訂單號碼（#開頭）
    或 customer_id 查詢訂單。
    回傳狀態、金額、配送資訊。
    用途：訂單追蹤、退貨查詢"
}
好的描述包含：接受什麼輸入、回傳什麼、何時用它、何時不用它。

情境思考：工具選擇出錯，第一步做什麼？

你有 analyze_content 和 analyze_document 兩個工具，Claude 經常選錯。你的第一步是？

B — 改 description 是第一步。低成本、高效果、直接解決根因。

更好的做法：重命名工具消除功能重疊，例如 analyze_content → extract_web_results。或者拆成更專精的工具：extract_data_points、summarize_content、verify_claim_against_source。
Task 2.2

Task 2.2: MCP 工具的結構化錯誤回應

泛用的 "Operation failed" 讓 agent 無法判斷該重試、放棄、還是換方案。

✗ 泛用錯誤
return {
  "isError": true,
  "content": "Operation failed"
}

// Agent 收到這個能做什麼？
// 不知道是暫時性還是永久性
// 不知道要不要重試
// 不知道有沒有部分結果
✓ 結構化錯誤
return {
  "isError": true,
  "content": {
    "errorCategory": "transient",
    "isRetryable": true,
    "description": "API timeout after 30s",
    "partialResults": [...3 筆已取得],
    "suggestion": "用更窄的日期範圍重試"
  }
}

// Agent 知道：可以重試、有部分結果、
// 該怎麼修改查詢
Python 四種錯誤類型範例 ▼ 
# 暫時性錯誤 → 可重試
{ "errorCategory": "transient", "isRetryable": True,
  "description": "Database connection timeout" }

# 驗證錯誤 → 修正輸入後可重試
{ "errorCategory": "validation", "isRetryable": True,
  "description": "日期格式錯誤，預期 YYYY-MM-DD" }

# 業務錯誤 → 不可重試，需要告訴用戶
{ "errorCategory": "business", "isRetryable": False,
  "description": "退款超過 $500 上限，需人工審核",
  "customerMessage": "此金額需要主管核准，已轉交處理" }

# 權限錯誤 → 不可重試
{ "errorCategory": "permission", "isRetryable": False,
  "description": "此 agent 無權存取財務資料" }

# ⚠️ 重要區分：
# 「存取失敗」（timeout）≠「合法空結果」（查詢成功但沒匹配）
# 空結果不是錯誤！不要設 isError: true
Task 2.3

Task 2.3: 工具分配策略 — 少即是多

給 agent 太多工具（18 個）反而降低選擇可靠性。每個 agent 只給它角色需要的 4-5 個。

你希望 Claude 怎麼使用工具？
→ tool_choice: "auto"（預設值）
Claude 可以選擇用工具或直接回覆文字。適合一般對話場景。
風險：Claude 可能跳過工具直接回答，造成回覆不準確。
→ tool_choice: "any"
保證 Claude 一定會呼叫某個工具，但由它選擇哪一個。
適合：多個 extraction schema 且文件類型未知時，保證結構化輸出。
→ tool_choice: {"type": "tool", "name": "extract_metadata"}
強制呼叫特定工具。適合流程中必須先跑某步驟的場景。
適合：必須先跑 extract_metadata 再跑 enrichment 的 pipeline。

Task 2.5

Task 2.5: 內建工具速查 — 什麼場景用什麼工具？

情境思考：你要在整個 codebase 中找出所有呼叫 processRefund() 的地方

哪個工具最適合？

B — Grep。搜尋檔案內容中的 pattern 就用 Grep。

速記：
Grep = 搜尋內容（函式名稱、錯誤訊息、import）
Glob = 搜尋檔案名（**/*.test.tsx）
Read = 讀完整檔案 / Edit = 精準修改
Bash = 只用在系統指令，有專用工具時不用 Bash
