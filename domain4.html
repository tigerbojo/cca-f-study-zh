Task 4.1

Task 4.1: 精確標準 vs 模糊指令 — 為什麼「保守一點」沒有用？

✗ 模糊指令（無效）
"Review this code and only report
 high-confidence findings.
 Be conservative."

// 結果：Claude 依然報一堆 false positive
// 因為「高信心」「保守」沒有具體標準
// Claude 對自己的每個 finding 都有信心
✓ 明確標準（有效）
"只報告以下類型的問題：
 - Bug：邏輯錯誤、null reference、off-by-one
 - 安全：SQL injection、XSS、auth bypass

 跳過：
 - 風格偏好（命名、排版）
 - 已知的 local pattern（團隊慣用寫法）
 - 只出現在測試中的問題"

// 結果：false positive 降 70%
// 因為 Claude 有清楚的判斷邊界
Task 4.2

Task 4.2: Few-shot 範例 — 最強大的 Prompt 技巧

當詳細指令仍產出不一致結果時，2-4 個精準的 few-shot 範例通常一次解決。

Prompt Few-shot 範例：Code Review Finding 格式 ▼ 
# 以下是 review finding 的範例格式：

## 範例 1（應該報告 — Bug）
location: src/cart.ts:42
issue: calculateTotal() 沒有處理空購物車，
       items.reduce() 在空陣列上回傳 undefined
severity: HIGH
fix: 加上 items.length === 0 的 early return

## 範例 2（不應該報告 — 風格偏好）
location: src/cart.ts:15
issue: 變數名 "x" 不夠描述性
decision: SKIP — 風格偏好，不是 bug

## 範例 3（應該報告 — 安全）
location: src/api/users.ts:78
issue: SQL query 直接用字串串接 user input
       → SQL injection 風險
severity: CRITICAL
fix: 改用 parameterized query

# Few-shot 的力量：
# - 展示了「格式」（location/issue/severity/fix）
# - 展示了「判斷」（什麼該報告、什麼該跳過）
# - Claude 會泛化到新的 pattern，不只 match 範例
Task 4.3

Task 4.3: tool_use + JSON Schema — 保證結構化輸出

用 tool_use 搭配 JSON Schema 是產出結構化資料最可靠的方式 — 100% 消除 JSON 語法錯誤。

Python 用 tool_use 強制結構化輸出 ▼ 
# 定義 extraction tool 的 JSON Schema
extract_invoice = {
    "name": "extract_invoice",
    "description": "從發票文件抽取結構化資料",
    "input_schema": {
        "type": "object",
        "required": ["vendor", "total"],
        "properties": {
            "vendor": { "type": "string" },
            "total": { "type": "number" },
            // ✓ nullable — 文件中可能沒有這項
            // 避免 Claude 為了滿足 required 而捏造值
            "po_number": { "type": ["string", "null"] },
            // ✓ enum + "other" 模式
            "category": {
                "enum": ["office", "travel", "software", "other"]
            },
            "category_detail": { "type": ["string", "null"] }
        }
    }
}

# 強制呼叫這個工具
response = client.messages.create(
    model="claude-sonnet-4-5-20250514",
    tools=[extract_invoice],
    tool_choice={"type": "tool", "name": "extract_invoice"},
    messages=[{"role": "user", "content": invoice_text}]
)
tool_use 消除語法錯誤，但不防語意錯誤。
JSON 格式保證正確 ✓ ｜但 total 可能不等於 line items 的加總 ✗
→ 語意驗證仍需要額外的 validation + retry loop
Task 4.5

Task 4.5: Batch API vs 即時 API — 什麼時候該用哪個？

你的工作流需要多快拿到結果？
→ Synchronous API（即時呼叫）。
Pre-merge check、PR review — 開發者在等結果。
Batch API 最長要 24 小時，完全不適合阻塞式工作流。
→ Message Batches API。省 50% 成本！
隔夜跑的技術債務報告、nightly test generation。
用 custom_id 關聯 request/response。失敗的用 custom_id 只重送失敗的文件。
→ Message Batches API + 提前提交。
每週稽核、月報。計算提交時間：如果 SLA 是 30 小時，每 4 小時提交一次（24hr batch + 6hr 緩衝）。
提示：先在小樣本上精煉 prompt，再 batch 處理大量。
Task 4.6

Task 4.6: 為什麼讓 Claude 自己 Review 自己寫的 Code 效果差？

✗ Self-review（同一 session）
User: "寫一個認證模組"
Claude: [產出 code...]

User: "現在 review 你剛寫的 code"
Claude: "看起來很好，設計合理..."

// 問題：Claude 保留了寫 code 時的
// 推理 context，不容易質疑自己的決策
// → 漏掉微妙的 bug
✓ 獨立 instance review
Session 1 (Generator):
  "寫一個認證模組" → [產出 code]

Session 2 (Reviewer): ← 全新 instance
  "Review 這段 code: [code]"
  "這裡有潛在的 timing attack..."

// 獨立 instance 沒有 generator 的
// 推理 context → 更客觀
// 大型 PR → per-file pass + integration pass

情境思考：14 檔案的 PR review 出了什麼問題？

你的單次 review 分析 14 個檔案，結果：有些檔案回饋詳細、有些很表面、明顯 bug 被遺漏、對相同 pattern 給出矛盾回饋。怎麼修？

A — 拆成聚焦的 pass。根因是注意力稀釋：同時處理太多檔案，每個都不夠深入。

B 錯：更大 context ≠ 更好的注意力品質。
C 錯：共識機制反而壓抑間歇性抓到的真正 bug。
D：轉嫁負擔給開發者，沒有改善系統。
