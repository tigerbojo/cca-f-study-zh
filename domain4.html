Domain 4: Prompt 工程與結構化輸出

佔分 20%
核心考點：精確標準勝過模糊指令、Few-shot 範例的強大作用、tool_use + JSON Schema 保證結構化輸出、Message Batches API 的 50% 省成本但 24 小時延遲、Self-review 的限制。

Task 4.1: 用明確標準設計 Prompt 以提升精確度、減少 False Positive

必考知識

  • 明確標準優於模糊指令：「只在宣稱行為和實際程式碼行為矛盾時標記」vs「檢查 comment 是否準確」
  • 「保守一點」或「只報告高信心」這類泛用指令無法改善精確度
  • 高 false positive 率會摧毀開發者對整個系統的信任

必考技能

  • 寫明確的 review 標準：哪些問題要報告（bug、安全）vs 跳過（小風格、local pattern）
  • 暫時停用高 false positive 的類別來恢復信任，同時改進該類別的 prompt
  • 用具體程式碼範例定義每個嚴重度等級

Task 4.2: 用 Few-shot Prompting 提升輸出一致性與品質

必考知識

  • Few-shot 範例是達到一致格式化、可行動輸出的最有效技巧
  • Few-shot 範例能展示模糊案例的處理方式（如工具選擇、測試覆蓋缺口）
  • Few-shot 讓模型泛化判斷到新模式，而非只匹配預設案例
  • 有效減少抽取任務中的幻覺（處理非正式量測值、多樣文件結構）

必考技能

  • 建立 2-4 個針對模糊情境的 few-shot 範例，展示選擇理由
  • 包含展示特定輸出格式（位置、問題、嚴重度、建議修正）的範例
  • 提供區分可接受 pattern vs 真正問題的範例，減少 false positive 同時保持泛化能力
  • 展示不同文件結構的正確抽取方式（inline citation vs 參考書目）

Task 4.3: 用 tool_use 和 JSON Schema 強制結構化輸出

必考知識

  • tool_use + JSON Schema = 最可靠的結構化輸出方法，消除 JSON 語法錯誤
  • tool_choice 差異：
    • "auto"：模型可能回傳文字而非呼叫工具
    • "any"：必須呼叫工具，但可選哪個
    • 強制指定：必須呼叫特定工具
  • Strict JSON Schema 消除語法錯誤，但不防止語意錯誤（如明細不等於總計、值放錯欄位）
  • Schema 設計：required vs optional、enum + "other" + detail string pattern

必考技能

  • 定義 extraction tool 的 JSON Schema 為 input parameter，從 tool_use response 抽取結構化資料
  • tool_choice: "any"：多個 schema 且文件類型未知時保證結構化輸出
  • 強制特定工具：tool_choice: {"type": "tool", "name": "extract_metadata"}
  • 來源文件可能沒有的資訊 → schema field 設為 optional（nullable），防止模型捏造值
  • 模糊案例加 "unclear" enum 值、可擴展分類加 "other" + detail field
考試重點：tool_use 消除的是語法錯誤（JSON 格式壞掉），但語意錯誤（值放錯欄位、數字不對）仍需額外驗證。required field 會迫使模型捏造不存在的資訊 → 用 nullable/optional。

Task 4.4: 實作驗證、重試和回饋迴圈以提升抽取品質

必考知識

  • Retry-with-error-feedback：在重試時附上具體驗證錯誤引導模型修正
  • 重試的極限：當資訊根本不在來源文件中時，重試無效
  • 回饋迴圈設計：追蹤哪些 code construct 觸發 finding（detected_pattern field）
  • 語意驗證錯誤 vs schema 語法錯誤（後者已被 tool_use 消除）

必考技能

  • 實作 follow-up request：包含原文件 + 失敗的抽取結果 + 具體驗證錯誤
  • 判斷重試是否有效：資訊不在文件中 → 無效；格式/結構錯誤 → 有效
  • 加 detected_pattern field 分析開發者 dismiss finding 的模式
  • 自我修正驗證：抽取 "calculated_total" 和 "stated_total" 比對差異

Task 4.5: 設計高效的批次處理策略

必考知識

  • Message Batches API：50% 成本節省、最長 24 小時處理、無延遲保證 SLA
  • 適合非阻塞、延遲容忍的工作（隔夜報告、每週稽核、夜間測試生成）
  • 不適合阻塞式工作流（pre-merge check）
  • Batch API 不支援單一請求內的多輪 tool calling
  • custom_id field 用於關聯 request/response

必考技能

  • 阻塞式 pre-merge check → synchronous API；隔夜分析 → batch API
  • 根據 SLA 計算 batch 提交頻率（如 4 小時窗口保證 30 小時 SLA）
  • 處理 batch 失敗：用 custom_id 只重新提交失敗的文件（如 chunking 超出 context 的文件）
  • 先在樣本集上精煉 prompt，再 batch 處理大量，提升首次成功率

Task 4.6: 設計多 Instance 和多 Pass Review 架構

必考知識

  • Self-review 的限制：模型保留了生成時的推理 context，不容易質疑自己的決策
  • 獨立 review instance（沒有先前推理 context）比 self-review 指令或 extended thinking 更能抓到微妙問題
  • Multi-pass review：拆成 per-file local analysis + cross-file integration pass → 避免注意力稀釋和矛盾 finding

必考技能

  • 用第二個獨立 Claude instance review 產生的程式碼（不帶 generator 的推理 context）
  • 大型多檔 review → per-file pass（local issue）+ integration pass（cross-file data flow）
  • 驗證 pass：模型在每個 finding 旁自報信心，做校準式 review routing
