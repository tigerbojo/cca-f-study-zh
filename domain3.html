Domain 3: Claude Code 設定與工作流

佔分 20%
核心考點：CLAUDE.md 的三層架構（user / project / directory）、.claude/rules/ 路徑規則、Slash Command vs Skill 的差異、Plan Mode 的使用時機、-p flag 的 CI/CD 用法。

Task 3.1: 設定 CLAUDE.md 的階層、作用域與模組化組織

必考知識

  • CLAUDE.md 三層架構：
    • 用戶級 ~/.claude/CLAUDE.md — 只影響自己，不進 version control
    • 專案級 .claude/CLAUDE.md 或根目錄 CLAUDE.md — 團隊共享
    • 目錄級 子目錄中的 CLAUDE.md — 特定目錄覆寫
  • @import 語法：引用外部檔案保持模組化（如針對每個 package import 不同標準檔）
  • .claude/rules/ 目錄：依主題分檔的規則，替代巨型 CLAUDE.md

必考技能

  • 診斷層級問題（如新同事沒收到指令 → 因為指令在 user-level 而非 project-level）
  • 用 @import 在每個 package 的 CLAUDE.md 選擇性引入相關標準
  • 將大型 CLAUDE.md 拆分到 .claude/rules/（如 testing.md、api-conventions.md）
  • 用 /memory 指令驗證載入了哪些記憶檔案，診斷跨 session 不一致行為

Task 3.2: 建立自訂 Slash Command 和 Skill

必考知識

  • 專案級 Command .claude/commands/ — 透過 version control 共享
  • 用戶級 Command ~/.claude/commands/ — 個人使用
  • Skill 在 .claude/skills/ 中搭配 SKILL.md，支援 frontmatter：context: fork、allowed-tools、argument-hint
  • context: fork：在隔離的 sub-agent context 中跑 skill，避免污染主對話
  • 個人 skill 自訂：在 ~/.claude/skills/ 用不同名稱建立個人變體

必考技能

  • 在 .claude/commands/ 建立團隊共享的 slash command
  • 用 context: fork 隔離產出大量輸出的 skill（如 codebase 分析）
  • 用 allowed-tools 限制 skill 執行時的工具存取權
  • 用 argument-hint 在沒給參數時提示開發者
  • 判斷：Skill（on-demand、任務導向）vs CLAUDE.md（always-loaded、通用標準）

Task 3.3: 用路徑規則做條件式載入

必考知識

  • .claude/rules/ 檔案搭配 YAML frontmatter paths 欄位含 glob pattern
  • 路徑規則只在編輯匹配檔案時載入 → 減少不相關 context 和 token 使用
  • Glob pattern 規則優於目錄級 CLAUDE.md → 適合跨多目錄的慣例（如散佈在 codebase 中的測試檔）

必考技能

  • 建立帶 YAML frontmatter 的路徑規則（如 paths: ["terraform/**/*"]）
  • 用 glob pattern 對任何位置的特定類型檔案套用慣例（如 **/*.test.tsx）
  • 當慣例需跨 codebase 套用時，選路徑規則而非子目錄 CLAUDE.md
考試重點：測試檔散佈在整個 codebase（如 Button.test.tsx 和 Button.tsx 同目錄）→ 用 .claude/rules/ + glob pattern 是最佳解，因為目錄級 CLAUDE.md 無法跨目錄套用。

Task 3.4: 判斷何時用 Plan Mode vs 直接執行

必考知識

  • Plan Mode：複雜任務、大規模變更、多種可行方案、架構決策、多檔修改
  • 直接執行：簡單、範圍明確的變更（如單一函式加一個驗證條件）
  • Plan Mode 允許安全地探索 codebase 和設計方案再動手，避免昂貴的返工
  • Explore subagent：隔離冗長的探索輸出，回傳摘要保護主對話 context

必考技能

  • 架構性任務 → Plan Mode（微服務重構、45+ 檔案 library 遷移）
  • 範圍明確的變更 → 直接執行（單檔 bug fix、加日期驗證條件）
  • 用 Explore subagent 做冗長的探索階段，避免 context window 耗盡
  • 組合使用：Plan Mode 做調查 + 直接執行做實作

Task 3.5: 迭代精煉技巧

必考知識

  • 具體的 input/output 範例是溝通預期轉換最有效的方式
  • TDD 迭代：先寫測試 → 分享測試失敗結果 → 引導漸進改善
  • Interview pattern：讓 Claude 提問，浮現開發者沒想到的考量
  • 多個互相影響的問題 → 一次訊息處理；獨立問題 → 逐一修正

必考技能

  • 提供 2-3 個具體 input/output 範例來釐清轉換需求
  • 先寫涵蓋預期行為、邊界案例和效能需求的測試套件
  • 在不熟悉的領域用 interview pattern 浮現設計考量（如 cache invalidation 策略）
  • 互相影響的問題 → 單一詳細訊息；獨立問題 → 逐一迭代

Task 3.6: 將 Claude Code 整合到 CI/CD Pipeline

必考知識

  • -p（或 --print）flag：在自動化 pipeline 中以非互動模式執行 Claude Code
  • --output-format json + --json-schema：在 CI 中產出結構化輸出
  • CLAUDE.md 提供專案 context（測試標準、fixture 慣例、review 標準）給 CI 中的 Claude Code
  • Session context 隔離：產生程式碼的 session 不適合 review 自己的變更 → 需要獨立 review instance

必考技能

  • 用 -p flag 在 CI 中跑 Claude Code，避免互動式輸入卡住
  • 用 --output-format json + --json-schema 產出可機器解析的結果，自動貼為 PR inline comment
  • 重新 review 時包含先前結果，指示只報告新的或仍未解決的問題（避免重複 comment）
  • 在 CLAUDE.md 中記錄測試標準和 fixture，提升測試產出品質
