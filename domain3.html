Task 3.1

Task 3.1: CLAUDE.md 三層架構 — 誰的指令覆蓋誰？

CLAUDE.md 有三層，搞混了 → 團隊成員收不到指令，或個人設定洩漏到 repo。

情境思考：新同事為什麼沒有收到 code review 規範？

你設定了嚴格的 code review 規範（一定要跑 lint、commit message 格式等）。但新入職的同事說 Claude Code 完全沒遵守。你的設定放在哪裡有問題？

A 是問題所在。~/.claude/CLAUDE.md 是用戶級設定，不進 version control，新同事的機器上根本沒有這個檔案。

修正：把團隊共享的規範移到專案級 .claude/CLAUDE.md 或 .claude/rules/，這樣 clone repo 就自動套用。
Task 3.2

Task 3.2: Slash Command vs Skill — 什麼時候用哪個？

Slash Command
.claude/commands/review.md
  → /review 即可呼叫
  → 簡單的 prompt template
  → 沒有 frontmatter 設定

~/.claude/commands/my-helper.md
  → 個人用，不進 git

適合：快速指令、固定 prompt
Skill（進階）
.claude/skills/analyze/SKILL.md
---
context: fork       ← 隔離執行
allowed-tools: [Read, Grep]  ← 限制工具
argument-hint: "file path"   ← 提示參數
---

適合：
- 產出大量輸出 → context: fork 隔離
- 需要限制工具存取權
- 需要參數提示
context: fork 讓 skill 在隔離的 sub-agent context 中執行，產出的大量文字不會污染你的主對話。適合 codebase 分析、brainstorming 等冗長輸出的場景。
Task 3.3

Task 3.3: 路徑規則 — 讓正確的 convention 自動套用在正確的檔案

YAML .claude/rules/testing.md 範例 ▼ 
# 檔案：.claude/rules/testing.md
---
paths:
  - "**/*.test.tsx"    # 所有 React 測試
  - "**/*.test.ts"     # 所有 TS 測試
  - "**/*.spec.ts"     # spec 格式測試
---

# 只有在編輯匹配的檔案時才載入這些規則
# → 減少不相關 context，省 token

測試撰寫規範：
- 使用 describe/it 結構
- mock 外部 API，不 mock database
- 每個 test case 用 AAA 模式 (Arrange-Act-Assert)
- 測試名稱描述行為，不描述實作

# 為什麼用 .claude/rules/ 而不是子目錄 CLAUDE.md？
# 因為 test 檔散佈在整個 codebase：
#   src/components/Button.test.tsx
#   src/api/auth.test.ts
#   src/utils/date.spec.ts
# 子目錄 CLAUDE.md 無法跨目錄套用！
Task 3.4

Task 3.4: Plan Mode vs 直接執行 — 什麼時候該思考，什麼時候直接動手？

你接下來要做的事情，複雜度如何？
→ 直接執行。
範圍明確、改動小、不需要探索。直接寫 code 比規劃更快。
例如：加一個日期驗證條件、修一個有 stack trace 的 bug。
→ 先 Plan Mode 調查，再直接執行。
需要先理解哪些檔案受影響、有什麼依賴。Plan Mode 探索 → 確認方案 → 切回直接執行實作。
用 Explore subagent 做冗長的探索，避免塞爆主 session 的 context。
→ 一定要 Plan Mode。
多種可行方案、架構決策、數十個檔案。不先規劃就動手 = 昂貴的返工。
Plan Mode 讓你安全探索 codebase 和設計方案，不會產生任何實際變更。
Task 3.6

Task 3.6: CI/CD 整合 — 讓 Claude Code 在 Pipeline 中自動跑

Bash CI/CD 中的 Claude Code 用法 ▼ 
# ✗ 錯誤：會卡在等待互動式輸入
claude "Review this PR for security issues"

# ✓ 正確：-p flag = 非互動模式
claude -p "Review this PR for security issues"

# ✓ 進階：結構化 JSON 輸出 → 自動貼為 PR comment
claude -p \
  --output-format json \
  --json-schema '{"type":"object","properties":{"findings":{"type":"array"}}}' \
  "Review the diff for bugs and security issues"

# 重要：產生 code 的 session ≠ review 的 session
# 同一個 session 自我 review 效果差
# → 用獨立的 Claude instance 做 review
CI/CD 三大重點：
1. -p flag 非互動模式（不加 = pipeline 卡死）
2. --output-format json + --json-schema 產出結構化結果
3. 用獨立 instance review，不要讓生成 code 的 session 自我 review
