Task 1.1

Task 1.1: Agentic Loop — Claude 怎麼自主工作？

Agentic Loop 是 Claude 自主完成任務的核心機制。理解這個迴圈，就理解了整個 Agent 架構的基礎。

綠色虛線 = 迴圈繼續（tool results 加入歷史後再送給 Claude）｜紅色 = 迴圈結束

Python Agentic Loop 實作範例 ▼ 
import anthropic

client = anthropic.Anthropic()
messages = [{ "role": "user", "content": "查詢客戶 #1234 的訂單狀態" }]
tools = [get_customer_tool, lookup_order_tool]  # MCP 工具定義

# === Agentic Loop ===
while True:
    response = client.messages.create(
        model="claude-sonnet-4-5-20250514",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    # 關鍵判斷點：檢查 stop_reason
    if response.stop_reason == "end_turn":
        # Claude 認為任務完成，跳出迴圈
        print(response.content[0].text)
        break

    if response.stop_reason == "tool_use":
        # Claude 要求使用工具，執行並回傳結果
        tool_block = response.content[-1]  # tool_use block
        result = execute_tool(tool_block.name, tool_block.input)

        # 把 Claude 的回應和工具結果都加入歷史
        messages.append({ "role": "assistant", "content": response.content })
        messages.append({
            "role": "user",
            "content": [{
                "type": "tool_result",
                "tool_use_id": tool_block.id,
                "content": result
            }]
        })
        # 迴圈繼續 → Claude 看到工具結果後決定下一步

情境思考：你的 agentic loop 哪裡有問題？

同事寫了一個 agentic loop，用以下方式判斷何時停止。哪個做法是正確的？

C 是正確的。stop_reason 是 API 層級的結構化信號，可靠度 100%。

A 是反模式：解析自然語言文字來判斷終止，不可靠且 fragile。
B 是反模式：任意迭代上限作為主要停止機制。（可以作為安全網，但不應是主要邏輯）
D 有微妙問題：Claude 可能同時回傳 text + tool_use，檢查「沒有 tool_use block」不等於 end_turn。
Task 1.2

Task 1.2: Coordinator-Subagent 多 Agent 架構

當任務太複雜，一個 agent 搞不定時，用 Hub-and-spoke 架構：一個 Coordinator 指揮多個專責 Subagent。

Hub-and-spoke 架構：所有通訊都經過 Coordinator，Subagent 之間不直接溝通

重要：Subagent 有獨立 context — 不會自動繼承 Coordinator 的對話歷史。你必須在 prompt 中顯式傳入所有需要的資訊。

情境思考：報告為什麼只涵蓋視覺藝術？

你的多 Agent 系統研究「AI 對創意產業的影響」，最終報告只涵蓋視覺藝術（數位藝術、平面設計、攝影），完全遺漏音樂、寫作和電影。每個 subagent 都正確完成了被分配的任務。問題出在哪？

B 是根因。Coordinator 把「創意產業」只分解成視覺藝術的三個子任務。所有 subagent 都在被分配的範圍內正確工作了 — 問題在於它們被分配了什麼。

這是 Task Decomposition 太狹窄的典型案例。修正方式：讓 Coordinator 的 prompt 指定研究目標和覆蓋標準，而非 step-by-step 子任務清單。
Task 1.3

Task 1.3: Subagent 的 Spawn 和 Context 傳遞

Python 平行 Spawn 多個 Subagent ▼ 
# Coordinator 在一個回應中發出多個 Task tool call → 平行執行
# 不是分成多個 turn 逐一發出（那樣是序列的）

# Agent 定義
search_agent = AgentDefinition(
    name="search",
    description="搜尋網路資料",
    allowed_tools=["web_search"],  # 只給它需要的工具
)

analysis_agent = AgentDefinition(
    name="analysis",
    description="分析文件內容",
    allowed_tools=["read_doc"],   # 職責分離
)

# Coordinator 的 allowedTools 必須包含 "Task"
coordinator = AgentDefinition(
    name="coordinator",
    allowed_tools=["Task"],  # 沒有 Task 就無法 spawn subagent
    system_prompt="""
    你是研究協調者。根據查詢需求：
    1. 分析需要涵蓋哪些面向
    2. 同時 spawn 多個 subagent 搜尋不同面向
    3. 收集結果後評估覆蓋度
    4. 有缺口就再 spawn 補充搜尋

    重要：subagent 看不到你的對話歷史，
    你必須在 prompt 中傳入所有需要的 context。
    """
)
平行 spawn 的關鍵：在 Coordinator 的一個回應中發出多個 Task tool call，它們就會平行執行。如果分成多個 turn 逐一發出，就變成序列執行，浪費時間。
Task 1.4 & 1.5

Task 1.4–1.5: 程式化強制 vs Prompt 指引 — 什麼時候該用哪個？

這是考試最愛考的概念。記住：Prompt 是機率性的，Hook 是確定性的。

✗ Prompt 指引（機率性）
system_prompt = """
重要：在處理任何訂單操作前，
你必須先呼叫 get_customer
驗證客戶身份。

永遠不要跳過這個步驟。
"""

# 問題：Claude「通常」會遵守，
# 但 12% 的案例中仍然跳過了
# → 錯誤退款、帳戶識別錯誤
Prompt 指令有非零失敗率。當錯誤有金融後果時，這不夠可靠。
✓ 程式化強制（確定性）
# Hook 攔截 tool call
def pre_tool_use(tool_name, args):
    if tool_name in ["lookup_order",
                     "process_refund"]:
        if not state.customer_verified:
            return {
                "blocked": True,
                "reason": "必須先完成客戶驗證"
            }
    return {"blocked": False}

# 結果：100% 保證順序
# 不可能跳過驗證步驟
Hook 提供確定性保證。無論 Claude 怎麼推理，程式碼都會攔截違規操作。

互動決策：你該用 Hook 還是 Prompt？

如果 agent 違反了這個規則，會有什麼後果？
→ 用 Hook / 程式化強制。
退款前必須驗證身份、超過金額閾值必須人工審核 — 這些不能容忍任何失敗率。
PostToolUse hook 或前置條件閘門提供 100% 保證。
→ 先試 Prompt 指引，觀察失敗率。
如果失敗率可接受（< 5%），prompt 夠用。如果太高，再加 hook。
例如：建議 agent 先查看最近訂單再推薦產品。
→ Prompt 指引就好。
例如：回覆時用正式語氣、先感謝客戶等。不需要程式化強制。
Task 1.6

Task 1.6: 任務分解策略 — Prompt Chaining vs 動態分解

Prompt Chaining（可預測任務）
適合：Code Review 14 檔案

Step 1: 逐檔分析 local issue
  → file1.py 的 finding
  → file2.py 的 finding
  → ...

Step 2: Cross-file integration pass
  → 跨檔資料流分析
  → 一致性檢查

為什麼有效：
每一步的範圍和輸出都可預測
動態分解（開放式任務）
適合：「幫 legacy codebase 加測試」

Step 1: 掃描專案結構
  → 發現 3 個模組

Step 2: 分析每個模組的依賴
  → 發現模組 B 依賴外部 API

Step 3: 根據發現調整計畫
  → 模組 B 需要 mock → 先做
  → 模組 A 最簡單 → 先測試

為什麼有效：
每一步根據發現動態決定下一步
Task 1.7

Task 1.7: Session 管理 — Resume vs 開新 Session

情境思考：你昨天用 Claude Code 分析了 codebase，今天同事改了 5 個檔案。你怎麼繼續？

你昨天在 session "refactor-auth" 中分析了認證模組的架構。今天同事推了 5 個 commit 修改了其中 3 個檔案。

B 最好。Resume 保留了昨天的 context（架構分析還是有效的），但你需要告訴 Claude 哪些檔案變了，讓它針對性重新分析，而不是全部重來。

A 的問題：Claude 不知道檔案被改了，可能基於過時的 tool results 做決策。
C 太浪費：大部分分析還是有效的，不需要從頭來。
D 是用在探索不同方案的場景（如比較兩種重構策略），不是這個情境。
Session 管理速記：
--resume — 先前 context 大致有效，只需更新部分內容
fork_session — 從同一基線探索不同方案（如 A 方案 vs B 方案）
開新 session + 注入摘要 — 先前 tool results 已過時，resume 反而有害
