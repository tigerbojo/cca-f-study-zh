Domain 1: Agentic 架構與編排

佔分 27% — 最高權重
核心考點：這個 Domain 考的是你對 Agentic Loop 生命週期的理解 — 怎麼用 stop_reason 控制迴圈、怎麼設計 Coordinator-Subagent 多 Agent 系統、以及何時該用程式化強制（Hooks）而非 Prompt 指引。

Task 1.1: 設計 Agentic Loop 進行自主任務執行

必考知識

  • Agentic Loop 生命週期：發送請求 → 檢查 stop_reason（"tool_use" vs "end_turn"）→ 執行工具 → 回傳結果
  • Tool results 被 append 到對話歷史，讓模型能推理下一步行動
  • Model-driven 決策（Claude 自己決定用哪個工具）vs 預設 decision tree（固定工具流程）

必考技能

  • 實作 agentic loop：stop_reason === "tool_use" 時繼續迴圈，"end_turn" 時終止
  • 每次迭代間將 tool results 加入 conversation context
  • 避免反模式：不要用自然語言解析來判斷終止、不要用任意迭代上限作為主要終止機制、不要檢查 assistant text 來判斷完成

Task 1.2: 用 Coordinator-Subagent 模式編排多 Agent 系統

必考知識

  • Hub-and-spoke 架構：Coordinator 管理所有 subagent 間的通訊、錯誤處理、資訊路由
  • Subagent 擁有獨立 context — 不會自動繼承 coordinator 的對話歷史
  • Coordinator 負責：任務分解、委派、結果聚合、動態選擇要呼叫哪個 subagent
  • 過度狹窄的任務分解風險：可能導致研究主題覆蓋不完整

必考技能

  • 設計 coordinator 動態分析需求、選擇要呼叫哪些 subagent（不是永遠跑完整流程）
  • 分配不同 subagent 負責不同子主題或來源類型，減少重複
  • 迭代精煉迴圈：coordinator 評估 synthesis 品質 → 發現缺口 → 重新委派 search/analysis → 重新 synthesis
  • 所有 subagent 通訊都經過 coordinator，確保可觀測性和一致的錯誤處理

Task 1.3: 設定 Subagent 的呼叫、Context 傳遞與 Spawn

必考知識

  • Task 工具是 spawn subagent 的機制，coordinator 的 allowedTools 必須包含 "Task"
  • Subagent context 必須顯式提供在 prompt 中 — 不會自動繼承、不會共享記憶
  • AgentDefinition 設定：描述、system prompt、工具限制
  • fork_session：從共享分析基線建立獨立分支，探索不同方案

必考技能

  • 在 subagent prompt 中直接包含前面 agent 的完整發現（例如把 web search 結果傳給 synthesis agent）
  • 用結構化格式分離內容與 metadata（source URL、文件名、頁碼）保留出處
  • 平行 spawn：在 coordinator 的一個回應中發出多個 Task tool call（而非分成多個 turn）
  • Coordinator prompt 應指定研究目標和品質標準，而非 step-by-step 指令

Task 1.4: 實作多步驟工作流的強制與交接模式

必考知識

  • 程式化強制（hooks、前置條件閘門）vs Prompt 指引：工作流順序保證
  • 當需要確定性合規（如身份驗證後才能做金融操作），prompt 指令有非零失敗率
  • 結構化交接協議：包含客戶詳情、根因分析、建議行動

必考技能

  • 實作程式化前置條件：在 get_customer 回傳已驗證 customer ID 前，封鎖 process_refund
  • 分解多面向客戶請求為獨立項目，平行調查後綜合統一回覆
  • 編制結構化交接摘要（customer ID、根因、退款金額、建議動作）給人類 agent
考試陷阱：當題目問「如何確保 agent 一定先驗證客戶身份」時，正確答案通常是程式化強制（Hook / 前置條件），而不是「在 system prompt 加強指示」或「加 few-shot 範例」。Prompt 方法是 probabilistic，不夠可靠。

Task 1.5: 用 Agent SDK Hooks 攔截 Tool Call 和正規化資料

必考知識

  • PostToolUse hook：攔截工具結果，在模型處理前做轉換
  • 攔截外發 tool call 以強制合規規則（如封鎖超過閾值的退款）
  • Hook 提供確定性保證，prompt 只提供機率性合規

必考技能

  • 用 PostToolUse hook 正規化不同 MCP 工具的異質資料格式（Unix timestamp、ISO 8601、數字狀態碼）
  • 攔截違反政策的 tool call（如退款超過 $500）並重導至替代工作流（如人工升級）
  • 當業務規則需要保證合規時，選擇 hook 而非 prompt

Task 1.6: 設計複雜工作流的任務分解策略

必考知識

  • 固定順序 pipeline（Prompt Chaining） vs 動態自適應分解
  • Prompt Chaining：將 review 拆成逐步驟（如逐檔分析 → 跨檔整合）
  • 自適應調查計畫：根據每步發現動態產生子任務

必考技能

  • 可預測的多面向 review → Prompt Chaining；開放式調查 → 動態分解
  • 大型 code review：拆成 per-file local analysis + 獨立的 cross-file integration pass，避免注意力稀釋
  • 開放式任務（如「幫 legacy codebase 加測試」）：先 mapping 結構 → 識別高影響區域 → 建立優先計畫

Task 1.7: 管理 Session 狀態、恢復與分叉

必考知識

  • --resume <session-name>：繼續特定的先前對話
  • fork_session：從共享分析基線建立獨立分支，探索不同方案
  • 恢復 session 後要告知 agent 哪些檔案有變更
  • 當先前 tool results 已過時，開新 session 注入摘要比 resume 更可靠

必考技能

  • 用 --resume 搭配 session name 跨工作階段繼續調查
  • 用 fork_session 建立平行探索分支（如比較兩種測試策略）
  • 判斷何時 resume（context 大致有效）vs 開新 session 注入摘要（tool results 已過時）
學習建議：Domain 1 佔 27%，是最高權重。務必熟記 stop_reason 的兩個值、Hub-and-spoke 架構、Hook vs Prompt 的選擇準則。建議實際用 Agent SDK 跑一次完整的 multi-agent flow。
